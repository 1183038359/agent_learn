1. 角色定位：Agent 算法工程师到底做什么

Agent 算法工程师不是“会把 LangChain 接起来”的工程师，而是能把大模型变成可控、可评估、可恢复、可上线的决策系统的人。

算法侧

理解 Transformer、推理采样、上下文窗口、embedding、rerank、规划搜索、强化学习、评估指标。能判断问题来自模型能力、提示设计、检索质量、工具接口还是状态机设计。

系统侧

把 agent 拆成模型调用、工具执行、记忆、状态、权限、观测、队列、缓存、重试、人工审核等模块。能处理延迟、成本、并发、失败恢复和数据一致性。

产品侧

为真实任务设计边界：什么时候自动做，什么时候问人，什么时候停止。能把成功率、完成时间、人工接管率、误操作率转化为可监控指标。

你需要形成的核心判断

  • 什么时候用 workflow，什么时候用 agent：如果路径固定，优先 workflow；如果需要动态决定下一步、使用工具、处理开放环境，才引入 agent。
  • 模型不是控制系统：LLM 负责语义判断和候选行动，代码负责权限、状态转换、校验、超时、幂等和回滚。
  • 可观测比“聪明”更重要：没有 trace、eval、日志和回放，就无法判断优化是否真实有效。
  • 安全是架构问题：prompt injection、工具越权、数据泄露、供应链工具风险，需要通过权限、隔离、策略、审计和用户确认一起解决。

2. 主流 Agent 能力地图

当前主流 agent 技术栈大致可以分为七层。学习时不要按框架名字死记，而要按系统层次理解：每一层解决什么问题、输入输出是什么、失败模式是什么。

3. 总体学习路线：28 到 36 周

下面的路线假设你每周投入 12 到 18 小时。如果你已有深度学习和后端经验，可以压缩到 20 到 24 周；如果基础薄弱，建议拉长到 40 周，尤其不要跳过评估与安全。

第 1-4 周
基础能力补齐：Python 工程、API 服务、异步任务、Transformer/embedding 基础、LLM 推理参数。产出一个最小聊天服务和一个 embedding 检索 demo。
第 5-8 周
提示、结构化输出与工具调用：掌握 system/developer/user message、JSON schema、function calling、错误重试、工具接口设计。产出一个能查天气、读文件、调用搜索 API 的基础 agent。
第 9-13 周
RAG 与知识系统：学习 chunking、hybrid search、rerank、query rewrite、citation、context packing、GraphRAG。产出一个可追溯引用的文档问答系统。
第 14-18 周
Agent 控制与规划：实现 ReAct、plan-and-execute、state graph、human-in-the-loop、reflection。用 LangGraph 或 OpenAI Agents SDK 做一个多步骤任务助手。
第 19-23 周
评估、观测与安全：建立任务集、自动评分、trace、错误归因、prompt injection 防护、工具权限和审计。产出 eval dashboard 和安全测试集。
第 24-30 周
生产化与优化：缓存、批处理、队列、限流、成本控制、模型路由、失败恢复、部署监控。把前面的 agent 包装成可部署服务。
第 31-36 周
作品集和专项突破：选择代码 agent、数据分析 agent、企业知识库 agent、浏览器操作 agent 或多 agent 研究助手之一，完成可演示项目、技术文档和评估报告。

4. 模块详解：每部分学什么、怎么练、多久

模块 0：Python、后端与工程基础

Agent 工程最终要落到可运行系统，因此 Python 基础、API 服务、异步 IO、数据库和测试是底座。

3 到 4 周

技术原理

  • HTTP 请求、鉴权、超时、重试、幂等、分页和速率限制。
  • 同步、异步、队列和后台任务的差异。
  • SQLite/Postgres 的基本建模，日志与配置管理。
  • 单元测试、集成测试、mock 外部 API。

实现训练

  • 用 FastAPI 写一个 `/chat` 接口，支持请求日志和错误码。
  • 实现一个 tool registry，把 Python 函数注册为可调用工具。
  • 用 pytest 给工具函数、API handler 和异常分支写测试。
  • 用 SQLite 保存会话、消息、工具调用结果和 trace id。

验收标准：你能解释一次用户请求如何从 HTTP 进入服务、调用模型、触发工具、写入日志、返回结果；服务异常时不会直接崩溃。

模块 1：LLM 与 Transformer 原理

Agent 的“脑”是大模型。你不一定要从零训练千亿模型，但必须理解模型为什么会犯错，以及推理参数怎样改变行为。

4 到 5 周

必学概念

  • tokenization、embedding、attention、position encoding、KV cache。
  • 预训练、SFT、RLHF/DPO、指令遵循、对齐。
  • temperature、top_p、max_tokens、stop、seed、logprobs。
  • 幻觉、上下文遗忘、长上下文退化和工具选择错误。

技术原理

Transformer 用 self-attention 计算 token 之间的相关性。推理时模型不是“查数据库”，而是根据上下文分布生成下一个 token。Agent 可靠性不能只靠提示词，需要外部工具、检索、校验和状态机共同约束。

实现训练

  • 手写一个简化 attention 计算，理解 Q/K/V。
  • 比较不同 temperature 对工具选择和答案稳定性的影响。
  • 测量长提示下关键信息放在开头、中间、结尾的命中率。

验收标准：你能解释为什么 LLM 需要 tool calling 才能安全执行外部动作，也能解释为什么 RAG 不是简单“把文档塞进 prompt”。

模块 2：Prompt、上下文工程与结构化输出

Prompt engineering 已经升级为 context engineering：重点是把任务、工具、状态、示例、检索片段和输出约束组织成稳定上下文。

2 到 3 周

技术原理

  • 指令分层：system/developer/user/tool 消息各自承担不同约束。
  • 结构化输出：JSON schema、枚举值、必填字段、类型校验。
  • 少样本示例：示例能显著影响格式、风格和决策边界。
  • 上下文打包：把最相关、最新、权限允许的信息放入上下文。

实现训练

  • 为同一任务写普通 prompt、few-shot prompt、JSON schema prompt，并比较失败率。
  • 写一个输出解析器：校验 JSON，失败时把错误反馈给模型重试。
  • 实现 context packer：按 token 预算选择系统指令、历史摘要、检索片段和工具结果。

验收标准：你能把一个自然语言需求转成稳定的输入协议和输出协议，而不是靠“请你严格遵守格式”。

模块 3：工具调用、MCP 与 Agent-Computer Interface

工具是 agent 影响世界的手。工具设计越清晰，agent 越稳定；工具权限越模糊，系统越危险。

3 到 4 周

必学概念

  • OpenAI/Anthropic 等模型的 function/tool calling 机制。
  • MCP 的 tools、resources、prompts、servers、clients。
  • Agent-Computer Interface：给 agent 暴露怎样的操作界面最可靠。
  • 工具参数 schema、幂等 key、dry-run、confirmation、audit log。

实现训练

  • 把搜索、文件读取、日历查询封装成三个工具。
  • 实现统一工具执行器：校验参数、执行、捕获异常、返回结构化结果。
  • 给高风险工具加 human approval，例如发送邮件、删除数据、下单。
  • 写一个最小 MCP server，让 agent 通过统一协议读取资源或调用函数。

验收标准：你能独立设计一个工具 API，让模型“容易选对、难以误用、出错可恢复、执行可追溯”。

模块 4：RAG、检索与知识系统

Agent 需要可靠知识。RAG 的核心不是向量库，而是“把正确证据以正确形式放进上下文，并让答案可验证”。

4 到 5 周

必学概念

  • chunk size、overlap、semantic chunking、metadata filter。
  • dense retrieval、sparse retrieval、hybrid search、reranker。
  • query rewrite、multi-query、HyDE、contextual compression。
  • citation、faithfulness、answer relevancy、context precision/recall。

实现训练

  • 对 PDF/Markdown/网页建立索引，保留标题、页码、URL、时间戳。
  • 实现 BM25 + embedding 混合召回，再用 reranker 重排。
  • 实现答案引用，每句话尽量能追溯到证据片段。
  • 用 RAGAS 或自建 eval 集评估检索和生成质量。

常见坑

  • chunk 过大导致噪声多，过小导致语义断裂。
  • 只看生成答案，不看召回证据是否包含正确答案。
  • 没有权限过滤，把不该看的文档放入上下文。
  • 引用只装饰答案，不能真实定位证据。

验收标准：给一个企业文档库，你能做出带引用、可评估、可解释失败原因的问答系统。

模块 5：规划、推理与 Agent 控制流

Agent 的难点不在“调用一次模型”，而在多步骤任务中如何决定下一步、何时停止、何时回滚、何时让人接管。

4 到 5 周

必学模式

  • ReAct：交替生成推理、行动和观察。
  • Plan-and-execute：先分解计划，再逐步执行。
  • Router：根据任务类型选择专家、工具或子流程。
  • Reflection/Critic：执行后自检、修正或重试。
  • Human-in-the-loop：在高风险、低置信或失败时请求人类决策。

实现训练

  • 手写一个 ReAct loop，限制最大步数和工具白名单。
  • 用 LangGraph 实现一个含 Plan、Search、Write、Review 节点的状态图。
  • 加入 failure policy：工具超时、结果为空、模型输出非法时如何恢复。
  • 实现 checkpoint：任意节点失败后可以从上一个状态继续。

验收标准：你能为复杂任务选择合适控制模式，并解释为什么不是所有事情都应该交给一个自由循环的 agent。

模块 7：记忆、状态与长上下文管理

Agent 需要记住目标、历史、用户偏好、任务进度和外部世界状态。但“全部塞进上下文”既贵又不稳定。

2 到 3 周

记忆类型

  • 短期记忆：当前会话上下文和工具观察。
  • 长期记忆：用户偏好、事实、历史任务摘要。
  • 工作记忆：当前计划、待办、变量、约束。
  • 情景记忆：过去案例和可复用解决方案。

实现方式

  • message history + summary buffer。
  • 结构化状态对象，例如 JSON/Pydantic。
  • 向量记忆，用 embedding 召回相关历史。
  • 事件日志，支持 replay 和 audit。

风险控制

  • 记忆写入要经过规则或模型判断，避免污染。
  • 区分事实、偏好、推测和临时计划。
  • 敏感信息需要过期、脱敏和权限检查。
  • 记忆召回要带来源，避免“凭空记得”。

验收标准：你能设计一个 agent 状态 schema，让任务能恢复、能解释、能限制上下文长度。

模块 8：评估、可观测与实验体系

Agent 的质量不能靠主观演示判断。你需要任务集、指标、trace、错误分类和回归测试。

3 到 4 周

指标体系

  • Task success rate：最终任务是否完成。
  • Step accuracy：每一步计划和工具选择是否正确。
  • RAG 指标：context precision、context recall、faithfulness。
  • Operational：延迟、成本、重试次数、人工接管率。
  • Safety：越权调用、敏感信息泄露、prompt injection 成功率。

实现训练

  • 构建 50 到 100 条离线任务集，覆盖常规、边界、恶意输入。
  • 保存每次运行的 prompt、模型输出、工具调用、观察和最终答案。
  • 用 LLM-as-judge 做初筛，但关键任务必须有人类标注校准。
  • 建立回归测试：改 prompt、换模型、换检索策略后自动比较指标。

验收标准：你能说清楚一次优化让哪个指标提升、是否牺牲了成本或安全，而不是只说“感觉更好了”。

模块 9：安全、权限与可靠性

Agent 会读取数据、调用工具、写入系统，因此必须按“默认不信任模型输出”的原则设计。

2 到 3 周

安全威胁

  • Prompt injection：文档或网页指示 agent 忽略规则。
  • Tool abuse：诱导 agent 调用危险工具或传入恶意参数。
  • Data exfiltration：把私有数据发送给外部工具或用户。
  • Indirect prompt injection：通过邮件、网页、PDF 等外部内容注入。

防护原则

  • 最小权限：工具只给完成任务所需权限。
  • 高风险操作确认：发送、删除、支付、权限修改必须审批。
  • 数据分级：不同来源和敏感级别进入不同上下文。
  • 沙箱隔离：代码执行、浏览器操作和文件系统访问必须隔离。

可靠性机制

  • 最大步数、预算和超时。
  • 工具返回结构化错误，让模型有机会恢复。
  • 幂等请求和补偿动作，避免重复执行。
  • 断点续跑、任务取消、人工接管和审计日志。

验收标准：你能为一个 agent 画出 threat model，并写出每个高风险工具的权限、确认和审计策略。

模块 10：生产化、MLOps 与成本优化

上线后的 agent 面临真实流量、预算、模型波动、依赖服务故障和用户长尾需求。

3 到 4 周

工程能力

  • 模型路由：简单任务用小模型，复杂任务用强模型。
  • 缓存：embedding cache、检索 cache、工具结果 cache、prompt cache。
  • 队列：长任务异步执行，前端显示进度和可取消状态。
  • 监控：tokens、延迟、错误率、重试、工具失败、用户接管。

优化策略

  • 减少不必要上下文，避免每步都塞完整历史。
  • 把确定性逻辑放到代码里，不让模型做简单规则判断。
  • 对工具结果做摘要，保留原始结果用于审计。
  • 批量处理 embedding 和离线评估，避免在线阻塞。

验收标准：你能把 demo 变成一个有日志、监控、限流、权限、回归测试和部署说明的服务。

模块 11：多 Agent、代码 Agent 与前沿方向

多 agent 不是越多越好。它适合职责清晰、信息隔离、需要并行或互评的任务。

3 到 4 周

多 Agent

  • 角色分工：planner、researcher、executor、critic。
  • 通信协议：消息格式、共享状态、终止条件。
  • 冲突处理：投票、仲裁、置信度、人工介入。

代码 Agent

  • 读仓库、定位文件、改代码、跑测试、解释 diff。
  • 需要沙箱、版本控制、测试策略和回滚机制。
  • 核心指标是通过测试、减少回归和最小修改范围。

前沿方向

  • Computer use、browser agent、GUI grounding。
  • 长期自主任务、异步 agent、个人记忆。
  • Agentic RAG、GraphRAG、tool learning、self-improvement。

验收标准：你能判断一个任务是否真的需要多 agent，并能证明多 agent 版本比单 agent 或 workflow 更好。

5. 核心代码实现与解释

下面的代码是“理解原理”的最小实现，不追求覆盖某个 SDK 的全部特性。真正项目里建议使用成熟 SDK 或框架，但你需要理解这些框架在替你做什么。

5.1 最小工具注册表

from dataclasses import dataclass
from typing import Any, Callable

@dataclass
class Tool:
    name: str
    description: str
    schema: dict[str, Any]
    fn: Callable[..., Any]
    risk: str = "low"

class ToolRegistry:
    def __init__(self) -> None:
        self._tools: dict[str, Tool] = {}

    def register(self, tool: Tool) -> None:
        if tool.name in self._tools:
            raise ValueError(f"duplicate tool: {tool.name}")
        self._tools[tool.name] = tool

    def specs_for_model(self) -> list[dict[str, Any]]:
        return [
            {
                "name": tool.name,
                "description": tool.description,
                "parameters": tool.schema,
            }
            for tool in self._tools.values()
        ]

    def execute(self, name: str, arguments: dict[str, Any]) -> dict[str, Any]:
        if name not in self._tools:
            return {"ok": False, "error": f"unknown tool: {name}"}

        tool = self._tools[name]
        try:
            # 生产环境应在这里做 JSON schema 校验、权限检查、超时和审计。
            result = tool.fn(**arguments)
            return {"ok": True, "tool": name, "result": result}
        except Exception as exc:
            return {"ok": False, "tool": name, "error": str(exc)}
代码解释：`Tool` 描述工具的名字、说明、参数 schema 和真实函数。`specs_for_model()` 把工具描述交给模型，让模型知道可以调用什么。`execute()` 是工具网关入口，生产环境必须加参数校验、权限判断、超时、重试和审计日志。

5.2 最小 Agent 循环

MAX_STEPS = 8

def run_agent(model, registry: ToolRegistry, user_goal: str) -> str:
    messages = [
        {"role": "system", "content": "You are a careful task agent. Use tools only when needed."},
        {"role": "user", "content": user_goal},
    ]

    for step in range(MAX_STEPS):
        response = model.chat(
            messages=messages,
            tools=registry.specs_for_model(),
        )

        if response.final_answer:
            return response.final_answer

        if not response.tool_call:
            messages.append({
                "role": "user",
                "content": "No valid final answer or tool call was produced. Decide the next safe step.",
            })
            continue

        call = response.tool_call
        tool_result = registry.execute(call.name, call.arguments)
        messages.append({
            "role": "assistant",
            "content": response.reasoning_summary,
            "tool_call": {"name": call.name, "arguments": call.arguments},
        })
        messages.append({
            "role": "tool",
            "name": call.name,
            "content": tool_result,
        })

    return "任务未在步数预算内完成，需要人工接管。"
代码解释：Agent loop 的本质是“模型决策，工具执行，观察结果，再决策”。`MAX_STEPS` 防止无限循环。工具结果要作为 tool message 放回上下文。真实系统里还要保存 trace、token 成本、每步状态和失败原因。

5.3 简化 RAG 检索与生成

import numpy as np

def cosine(a: np.ndarray, b: np.ndarray) -> float:
    return float(a @ b / (np.linalg.norm(a) * np.linalg.norm(b) + 1e-9))

class MiniRAG:
    def __init__(self, embed_model, llm):
        self.embed_model = embed_model
        self.llm = llm
        self.rows: list[dict] = []

    def add_document(self, doc_id: str, chunks: list[str]) -> None:
        for i, text in enumerate(chunks):
            vector = self.embed_model.embed(text)
            self.rows.append({
                "doc_id": doc_id,
                "chunk_id": i,
                "text": text,
                "vector": vector,
            })

    def retrieve(self, query: str, k: int = 5) -> list[dict]:
        q = self.embed_model.embed(query)
        scored = [
            (cosine(q, row["vector"]), row)
            for row in self.rows
        ]
        return [row for _, row in sorted(scored, reverse=True)[:k]]

    def answer(self, query: str) -> dict:
        evidence = self.retrieve(query)
        context = "\n\n".join(
            f"[{row['doc_id']}#{row['chunk_id']}]\n{row['text']}"
            for row in evidence
        )
        prompt = f"""
        只根据证据回答问题。如果证据不足，回答“证据不足”。
        回答中必须引用证据编号。

        问题：{query}
        证据：
        {context}
        """
        return {
            "answer": self.llm.generate(prompt),
            "evidence": [
                {"doc_id": row["doc_id"], "chunk_id": row["chunk_id"]}
                for row in evidence
            ],
        }
代码解释：这段代码展示 RAG 的最小闭环：文档切块、embedding、相似度召回、构造带证据编号的 prompt、生成带引用答案。生产环境要补 BM25、rerank、权限过滤、去重、上下文压缩、答案事实性评估。

5.4 状态图思维的伪代码

class AgentState(dict):
    """保存 goal、plan、evidence、draft、errors、approval 等字段。"""

def plan_node(state: AgentState) -> AgentState:
    state["plan"] = make_plan(state["goal"])
    return state

def search_node(state: AgentState) -> AgentState:
    state["evidence"] = search_documents(state["plan"])
    return state

def write_node(state: AgentState) -> AgentState:
    state["draft"] = write_answer(state["goal"], state["evidence"])
    return state

def review_node(state: AgentState) -> str:
    report = evaluate_draft(state["draft"], state["evidence"])
    state["review"] = report
    if report["safe"] is False:
        return "human_approval"
    if report["score"] < 0.75:
        return "search"
    return "finish"
代码解释：状态图把 agent 拆成节点。每个节点只做一件事，读写同一个 `state`。`review_node` 返回下一条边：继续搜索、请求人工审批或完成。LangGraph 这类框架会在此基础上提供 checkpoint、恢复、interrupt 和 trace。

5.5 最小评估脚本

def exact_or_judge_score(example: dict, prediction: dict, judge_llm) -> dict:
    if "expected_answer" in example:
        exact = example["expected_answer"].strip() in prediction["answer"]
    else:
        exact = None

    judge_prompt = f"""
    请判断回答是否完成任务，返回 JSON：
    {{
      "success": true/false,
      "reason": "...",
      "safety_issue": true/false
    }}

    任务：{example["input"]}
    回答：{prediction["answer"]}
    证据：{prediction.get("evidence", [])}
    """
    judged = judge_llm.generate_json(judge_prompt)
    return {
        "id": example["id"],
        "exact_match": exact,
        "success": judged["success"],
        "safety_issue": judged["safety_issue"],
        "reason": judged["reason"],
    }

def run_eval(agent, dataset: list[dict], judge_llm) -> list[dict]:
    results = []
    for example in dataset:
        prediction = agent.run(example["input"])
        results.append(exact_or_judge_score(example, prediction, judge_llm))
    return results
代码解释：评估脚本需要把输入、输出、证据、分数和原因保存下来。LLM-as-judge 可以提高效率，但不能完全替代人工标注。关键业务要抽样人工复核，并检查 judge 本身的偏差。
